# 干擾設計與 AI 協作方式說明

## 一句話定位

九宮格 AI Vision Benchmark 是一個「可控資料生成 + 畫面捕捉 + 多演算法判斷 + 分析圖表 + 雙紀錄匯出 + AI 回饋校正」的小型實驗場。

它用簡單九宮格示範一個重要觀念：

> AI 應用落地不是只問模型答案，而是要設計資料、觀測、判斷、紀錄與校正流程。

## 為什麼要做干擾

沒有干擾時，顏色辨識很容易。

真正值得教的是：

```text
畫面被遮擋、偏移、加雜訊、被色塊污染後
系統如何定義正確答案？
演算法如何判斷？
紀錄如何支援後續 AI 分析？
```

所以干擾不是裝飾，而是用來製造可分析的錯誤情境。

## 兩種任務定義

### 任務 A：底圖還原壓力測試

這類任務的正確答案永遠是底圖。

```text
底圖 = 真實狀態
遮罩 / 色塊 = 干擾
捕捉框 = 掃描器實際觀測區域
```

即使遮罩在捕捉框內超過 50%，正確答案仍然是底圖。

這對應到實務情境：

```text
產線燈號本來是綠燈
畫面上有反光、貼紙、遮擋
系統仍然要判斷燈號原本狀態
```

目前拆成兩個版本：

| 測試 | 遮罩占 ROI | 正確答案 | 教學目的 |
|---|---:|---|---|
| 壓力 A | 大於 50% | 底圖色 | 測強干擾下是否被遮罩騙走 |
| 壓力 B | 小於 50% | 底圖色 | 測弱干擾下是否能穩定回到底圖 |

### 任務 B：雙色比例測試

這類任務的正確答案不是底圖，而是面積多數色。

```text
每格隨機產生兩種顏色
比例介於 20% 到 80%
正確答案 = 面積占比較多的顏色
```

這個測試用來說明：

```text
如果任務是「判斷目前可見畫面主色」
正確答案就應該依可見比例決定
```

## 為什麼要分開

同樣是九宮格，同樣是顏色，正確答案可能不同。

差別不在技術，而在任務定義：

```text
底圖還原任務：答案是底圖
可見比例任務：答案是多數色
```

這是 AI 落地很重要的一課：

> Ground Truth 不是天生存在，而是由業務目標、感測方式與驗證規則一起定義。

## 演算法角色

| 方法 | 直覺 | 容易失敗的地方 |
|---|---|---|
| 中央區域平均 | 看捕捉框中央的平均 RGB | 遇到遮罩色塊時容易被拉偏 |
| 整格平均 RGB | 看整格平均 | 邊框、遮罩、雙色混合會產生中間色 |
| 主要顏色投票 | 每個 pixel 投給最接近的 palette 顏色 | 如果遮罩占多數，可能忠實投給遮罩 |
| 綜合加權 | 把多個方法依權重做決策 | 權重設定錯時會放大錯誤方法 |

## 決策權重設計

為了讓版面高度穩定，也讓資料格式更一致，四種演算法都用同一個權重框架呈現。

| 模式 | 中央區域平均 | 整格平均 RGB | 主要顏色投票 | 教學意義 |
|---|---:|---:|---:|---|
| 中央區域平均 | 100 | 0 | 0 | 單一方法是綜合決策的特例 |
| 整格平均 RGB | 0 | 100 | 0 | 只相信整格平均 |
| 主要顏色投票 | 0 | 0 | 100 | 只相信 pixel 投票 |
| 綜合加權 | 33 | 33 | 34 | 三種方法共同決策，且可調整 |

## 雙紀錄設計

### Sheet 1：capture_log

每次掃描都寫入。

用途：

- 保留每格最後辨識結果。
- 保留干擾條件。
- 保留演算法與 accuracy。
- 支援整體統計。

### Sheet 2：weighted_decision_log

只有綜合加權模式寫入。

用途：

- 保留三個基礎演算法各自判斷。
- 保留權重比例。
- 保留最後決策分數。
- 支援後續 AI 分析權重是否合理。

## 分析圖表設計

分析圖表不是另一個演算法，而是把紀錄先轉成「人和 AI 都容易讀」的摘要。

目前保留四種圖表：

| 圖表 | 來源 | 用途 |
|---|---|---|
| 演算法準確率 | `capture_log.algorithm` | 比較不同辨識方法在目前資料集中的可靠度 |
| 干擾條件準確率 | `capture_log.interference_key` | 找出最容易造成錯判的干擾 |
| 真實顏色命中率 | `capture_log.truth_color_name` | 檢查某些顏色是否特別容易被誤判 |
| 錯判對照 | truth -> detected | 讓 AI 分析錯誤路徑，例如紅色常被判成粉色 |

這一層的教學重點是：

```text
先看圖表找現象
再回紀錄找原因
最後讓 AI 建議下一輪策略
```

分析圖表提供兩種顯示模式：

| 模式 | 用途 |
|---|---|
| 單色版 | 適合正式報告、README、截圖 |
| 多彩版 | 有顏色語意的資料會盡量對應原色，例如粉色命中率用粉色，錯判路徑用 truth -> detected 漸層 |

## 操作案件對應

| 按鈕 | 對應案件 | 教學意義 |
|---|---|---|
| 開始 | 啟動定時產生與定時掃描 | 建立連續觀測資料 |
| 結束 | 停止定時任務 | 保留狀態，進行觀察與討論 |
| 產生一次 | 只刷新 Ground Truth | 控制單一測試樣本 |
| 掃描一次 | 只做一次辨識 | 比較演算法與干擾條件 |
| 改變版面配色 | 標準、實驗綠、活力橘 | 示範同功能在不同教學場景的呈現 |
| ＋ / － 視窗縮放 | 調整整個介面 | 示範畫面尺度如何影響觀察、展示與 Computer Use |

## 鍵盤熱鍵

| 熱鍵 | 動作 | 用途 |
|---|---|---|
| `S` | 開始 | 啟動連續產生與掃描 |
| `E` | 結束 | 停止連續流程 |
| `G` | 產生一次 | 刷新 Ground Truth |
| `C` | 掃描一次 | 執行單次辨識 |
| `T` | 切換配色 | 標準、實驗綠、活力橘輪流切換 |
| `+` / `-` | 放大 / 縮小 | 調整整個操作介面 |
| `A` | 分析圖表 | 進入統計觀察頁 |
| `H` / `?` | 專案說明 | 進入教學文件頁 |

## AI 協作方式

這個 demo 可以教出「不同 AI 角色」：

| AI 類型 | 在本專案中的角色 | 實務對應 |
|---|---|---|
| 模擬資料 AI | 產生九宮格、比例、干擾條件 | 合成資料、測試案例生成 |
| 辨識 / 運用 AI | 看畫面、判斷顏色或文字 | OCR、Vision、Computer Use |
| 分析校正 AI | 讀 CSV，分析錯誤與建議權重 | 評估、監控、策略調整 |
| 部署運行 AI | 把流程變成可重複服務 | Runtime、API、Agent Workflow |

這也呼應 NVIDIA AI 生態系的方向：NVIDIA 官方把企業 AI 拆成 NIM / NeMo / Blueprints、RAG、Agent、Video Analytics、Robot Simulation、AI Factory 等不同層次與工作流。換句話說，AI 生態不是單一模型，而是從資料、模型、工作流、部署到回饋的完整系統。
